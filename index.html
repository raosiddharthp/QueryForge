QueryForge – Multi-Query Optimization Engine for Enterprise RAG

1. Description

QueryForge is an open-source, local-first optimization engine for enterprise Retrieval-Augmented Generation (RAG) systems. It automatically generates multiple query variants for a given user question (decomposition, HyDE, step-back prompting, rewriting), runs parallel retrieval across different strategies (vector, BM25, metadata), fuses and reranks results, and evaluates end-to-end performance to recommend the optimal configuration for any specific dataset. Designed to solve the #1 production RAG failure mode — poor retrieval quality — QueryForge turns manual tuning into automated, reproducible science while running entirely offline for data privacy.

2. Executive Summary

QueryForge addresses the core challenge in production RAG: single-query retrieval often misses relevant context due to ambiguous phrasing or scattered information. Manual tuning of chunk size, top-k, or query rewriting is time-consuming and rarely optimal. QueryForge automates the discovery of the best multi-query + retrieval strategy mix, delivering measurable improvements in recall, answer accuracy, and latency. Built for local execution with open-source components, it enables enterprises to optimize RAG performance on proprietary data without cloud dependency. The system serves as both a practical MLOps tool and a demonstration of systematic LLM engineering applied to retrieval optimization.

3. Business Strategy

3.1 Strategic Value Proposition

QueryForge reduces RAG tuning effort by 80–90% and improves retrieval recall by 20–40% on average, directly translating to higher answer quality in enterprise knowledge systems (support, compliance, research). It enables reproducible, metric-driven optimization rather than guesswork, making RAG deployments more reliable and cost-effective. Primary value drivers: faster time-to-production RAG, reduced hallucination rates, and improved ROI on LLM investments.

3.2 Regulatory Strategy

All processing occurs locally with no external API calls. Input documents and evaluation queries remain on-device. Open-source code allows security audits. Future extensions can add configurable redaction for sensitive content without changing the core offline model.

5. Design and Architecture

5.1 Phase A: Vision

Automate and systematize RAG retrieval optimization to make production-grade performance achievable without manual trial-and-error.

5.2 Phase B: Business

Core capabilities: multi-query generation, parallel retrieval, fusion/reranking, automated evaluation, config recommendation. Success metrics: retrieval recall@10 improvement, answer accuracy gain, time saved vs manual tuning.

5.3 Phase C: Information

Input: document corpus + evaluation queries (with ground-truth relevant passages). State tracks experiment configurations, retrieval results, metrics.

5.4 Phase D: Technology

  • Query Generation: LLM-driven rewriting (decomposition, HyDE, step-back)
  • Retrieval: Chroma (vector), BM25, metadata filters
  • Fusion/Reranking: Reciprocal rank fusion + optional bge-reranker
  • Evaluation: Recall@k, MRR, LLM-as-judge accuracy
  • Frontend: Streamlit dashboard with experiment comparison

7. Optimization Pipeline

7.1 Core Components

  • Query Rewriter: Generates variants (decomposition, HyDE, step-back, synonym expansion)
  • Retrieval Executor: Runs parallel searches (vector, BM25, metadata)
  • Fusion Engine: Reciprocal rank fusion + optional reranker
  • Evaluator: Computes recall@k, MRR, answer accuracy
  • Recommender: Selects best-performing configuration

7.2 Experiment Workflow

Grid search across parameters with automated evaluation and ranking.

7.3 Decision Matrix

Configurations scored on weighted metrics (recall primary, latency secondary). Top 3 recommended with trade-off analysis.

9. The Model Lifecycle (MLOps Focus)

QueryForge emphasizes MLOps principles in its model lifecycle, treating retrieval and query rewriting as production assets. Following the App > Agent > Platform > Model > Infrastructure pattern:

  • Model Selection: Default Llama 3.1 8B for query rewriting; configurable via Ollama
  • Prompt Management: Version-controlled templates for each query strategy
  • Evaluation: Rigorous offline testing on domain-specific query sets with metrics (recall@k, MRR, answer correctness via LLM-as-judge)
  • Continuous Improvement: Experiment tracking and community contributions for new strategies
  • Deployment: Local inference with model quantization for efficiency
