QueryForge: Automated Multi-Query Optimization for Enterprise RAG Systems

Siddharth Rao · Architecture Portfolio · March 2026
Abstract

Single-vector retrieval — the standard implementation in enterprise RAG deployments — fails predictably on queries that require reasoning across multiple documents, temporal context, or entity disambiguation. This document describes QueryForge, an open-source engine that intercepts queries before retrieval, classifies their structural complexity, decomposes multi-hop queries into atomic sub-queries, executes multiple retrieval strategies in parallel, and merges results using Reciprocal Rank Fusion. The system logs per-strategy scores and outputs configuration recommendations calibrated to the observed query distribution. The primary target is engineering teams at enterprise search vendors — Glean, Notion AI, Confluence AI, and similar — who have shipped RAG and need systematic tools to diagnose and improve retrieval quality without replacing existing infrastructure.

Keywords: retrieval-augmented generation, query decomposition, hybrid retrieval, reciprocal rank fusion, BM25, cross-encoder reranking, enterprise search

§1 Motivation

The majority of enterprise search platforms deployed RAG capabilities between 2023 and 2024. The implementation pattern is largely uniform: documents are chunked, embedded into a vector store, and retrieved via cosine similarity against an embedded query. This works acceptably for factual single-hop lookups and degrades sharply on anything more structurally complex.

The engineering teams responsible for these systems face a consistent set of difficulties. Retrieval failure is difficult to attribute — it is unclear whether poor results originate from chunking strategy, embedding model choice, query formulation, retrieval depth, or the absence of reranking. Configuration is typically set once at build time on a small set of test queries and never revisited as the corpus and query distribution evolve. There is no instrumentation.

Three developments in 2024–2025 made a systematic approach tractable. First, open-source cross-encoder models — specifically the ms-marco-MiniLM family — reached quality sufficient for production reranking at negligible inference cost. Second, asyncio-based orchestration frameworks made running four retrieval strategies concurrently practical; total wall time is bounded by the slowest single strategy rather than their sum. Third, lightweight LLM-based query classification became fast enough (~200ms) to run on every incoming query, enabling adaptive strategy selection before retrieval begins.

Design constraint

QueryForge is designed as a layer on top of existing infrastructure, not a replacement. It requires no migration of vector stores, embedding models, or generation components. The integration surface is a single REST endpoint or SDK wrapper around the existing retrieval call.

§2 Problem Statement

The core failure mode is straightforward. A user query is embedded into a single vector. The nearest chunks in the vector store are retrieved. If the answer to the query is distributed across multiple documents with no overlapping terminology, no single chunk will have high similarity to the query embedding, and the retrieval will return the nearest available approximation with high confidence. The generation step then produces a fluent, confident, incorrect answer.

This failure mode is not an edge case. In a corpus typical of an enterprise knowledge base — policy documents, legal addenda, SOPs, HR policies, runbooks — the majority of non-trivial employee queries require reasoning across two or more documents. 2.2 The Structural Failure — A Worked Example

Consider the query: "What is our approval process for vendor contracts over $50K with non-standard payment terms?" The answer is distributed across three documents with no overlapping terminology between them.

# Doc A, Doc B, Doc C: not in top-3 · no shared terminology with query
# outcome: fluent, confident, incomplete answer
Key observation

The retrieval does not fail with a low confidence score. It returns a result with similarity 0.84 — which the generation layer treats as a valid retrieval. The error is silent. The answer sounds correct.

§3 Architecture

QueryForge wraps the retrieval step in an existing RAG pipeline. The integration requires no changes to the vector store, embedding model, or generation component. The system exposes a REST endpoint that accepts a query string and returns a ranked result set along with a configuration recommendation.

3.1 Pipeline

Incoming queries pass through five sequential stages. Stages 3–5 execute in parallel via asyncio; total latency is bounded by the slowest strategy rather than their sum. Simple queries classified as single-hop skip Stage 2 entirely.

3.2 Retrieval Strategies

Four strategies run concurrently. The classifier's output determines which strategies are active; single-hop queries use dense only. The alpha weight in hybrid retrieval is tuned dynamically based on the detected query type — higher semantic weight for conceptual queries, higher term weight for entity-heavy or numeric queries.

dense vector
text-embedding-3-small
Cosine similarity over query and document embeddings. Default strategy for all query types. Strong on paraphrase tolerance; weak on exact terminology and numeric constraints.
single-hop · semantic similarity
sparse BM25
rank-bm25
Term-frequency / inverse-document-frequency scoring. Strong on exact entity names, contract numbers, and numerical values. Complements dense retrieval in hybrid mode.

hybrid
α·dense + (1-α)·BM25
Weighted linear combination. α is set per query type: 0.7 for conceptual, 0.4 for entity-heavy, 0.5 for temporal. Default strategy for comparative and multi-hop queries.
mixed queries · default for complex types
cross-encoder
ms-marco-MiniLM-L-6-v2
Two-stage: retrieve top-20 candidates with hybrid, rerank with a cross-encoder that jointly encodes query and document. Highest latency (+600ms); used selectively for complex queries where precision is the priority.
precision-critical · complex multi-hop

LLM generates 2–5 atomic sub-queries from a complex input. Independent dense retrieval runs per sub-query. Candidate sets are merged before fusion. Only strategy capable of surfacing answers distributed across documents with no overlapping terms.
multi-hop · cross-document synthesis
HyDE

LLM generates a hypothetical answer. That answer is embedded and used as the query vector. Improves recall when query phrasing differs substantially from how the answer is written in the corpus.

3.3 Result Fusion

Results from all active strategies are merged using Reciprocal Rank Fusion (RRF). The formula assigns each candidate a score of Σ 1/(k + rank_i) where k=60 and the sum is over all strategy result lists in which the candidate appears. Documents appearing in multiple strategy results are promoted; the scheme is robust to score scale differences between dense and BM25 outputs.
